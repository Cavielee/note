## 事务

满足 ACID 特性的一系列操作。在数据库中可以通过 Commit 提交事务，也可以使用 Rollback 进行回滚。

### ACID

![img](https://github.com/Cavielee/Interview-Notebook/raw/master/pics/09e6e8d4-4d83-4949-b908-6d6b4c2fd064.png)

#### 原子性（Atomicity）

事务被视为不可分割的最小单元，要么全部提交成功，要么全部失败回滚。

#### 一致性（Consistency）

事务执行前后保持一致性状态。在一致性状态下，所有事务对一个数据的读取结果都是相同的。

#### 隔离性（Isolation）

一个事物所做的修改在最终提交以前，其他事务是不可见的。

#### 持久性（Durability）

一旦事务提交，则其所做的修改将会永远保存到数据库中。即使系统发生崩溃，事务执行的结果也不能丢失。可以通过数据可的备份和恢复来保证持久性。



## 并发一致性问题

当多个事务以串行的形式进行时，就确保了隔离性，不会导致并发一致性问题。

可以通过锁的形式，把事务并发变为串行进行。或者使用数据库定义的事务隔离级别来解决。

### 丢失数据

事务T1和事务T2在事务结束前都对同一个数据进行修改，导致其中一个事务中的修改操作被其他事务的修改操作所覆盖。

![img](https://github.com/Cavielee/Interview-Notebook/raw/master/pics/88ff46b3-028a-4dbb-a572-1f062b8b96d3.png)

### 脏读

事务T1读取了事务T2的修改后的数据，随后事务T2 Rollback，导致T1读取了不应该读取的数据。

![img](https://github.com/Cavielee/Interview-Notebook/raw/master/pics/dd782132-d830-4c55-9884-cfac0a541b8e.png)

### 不可重复读

事务T2读取一个数据，事务T1对该数据进行修改，如果T2再次读取该数据则结果与第一次读取的结果不同。

![img](https://github.com/Cavielee/Interview-Notebook/raw/master/pics/c8d18ca9-0b09-441a-9a0c-fb063630d708.png)

### 幻读

事务T1读取事务T2插入的新数据，T2进行 Rollback，则T1再次读取时，没有该数据。

## 封锁

由于修改数据时需要获取锁、检查锁、释放锁等操作，会消耗性能，因此应该尽量只锁定需要修改的那部分数据，而不是全部资源。这样可以增加并发程度。

Mysql 支持行锁和表锁两种。



### 排它锁和共享锁

* 排它锁（Exclusive），简称 X 锁，又称写锁。

  一个事务对数据 A 加了 X 锁，就可以对 A 进行读取和更新。加锁期间其他事务不能对 A 加任何锁。

* 共享锁（Shared），简称 S 锁，又称读锁。

  一个事务对数据 A 加了 S 锁，就可以对 A 进行读取，但不能进行更新。加锁期间其他事物只能对 A 加 S 锁，不能加 X 锁。



锁的兼容关系如下：

| -    | X    | S    |
| ---- | ---- | ---- |
| X    | No   | No   |
| S    | No   | Yes  |



### 意向锁

意向锁（Intention Lock）是一个表锁。用来表示一个事务想要在某个数据上加 X 锁或 S 锁。

1. 一个事务在获得某个数据行对象的 S 锁之前，必须先获得 IS 锁或者更强的锁；
2. 一个事务在获得某个数据行对象的 X 锁之前，必须先获得 IX 锁。

各种锁的兼容关系如下：

| -    | X    | IX   | S    | IS   |
| ---- | ---- | ---- | ---- | ---- |
| X    | No   | No   | No   | No   |
| IX   | No   | Yes  | No   | Yes  |
| S    | No   | No   | Yes  | Yes  |
| IS   | No   | Yes  | Yes  | Yes  |



## 封锁协议

### 一级封锁协议

事务 T 要修改数据 A 时必须加 X 锁，直到 T 结束才释放锁。

可以解决丢失修改问题，因为不能同时有两个事务对同一事务进行修改，那么一个事务的修改就不会被覆盖。

| T1          | T2               |
| ----------- | ---------------- |
| lock-x(A)   |                  |
| read A=20   |                  |
|             | lock-x(A)        |
|             | wait（等待锁）   |
| write A=19  |                  |
| commit      |                  |
| unlock-x(A) |                  |
|             | obtain（获取锁） |
|             | read A=19        |
|             | write A=21       |
|             | commit           |
|             | unlock-x(A)      |

### 二级封锁协议

在一级的基础上，要去读取数据 A 时必须要加 S 锁，读取完马上释放 S 锁。

由于一级协议中，如果一个事务要对数据 A 进行修改，会加 X 锁。其他事务若要读取数据 A 则要先加 S 锁，但数据 A 已经加了 X 锁，所以无法加 S 锁，所以解决脏读问题。

| T1          | T2                   |
| ----------- | -------------------- |
| lock-x(A)   |                      |
| read A=20   |                      |
| write A=19  |                      |
|             | lock-s(A)            |
|             | wait（等待释放 X锁） |
| rollback    |                      |
| A=20        |                      |
| unlock-x(A) |                      |
|             | obtain（获取 S 锁）  |
|             | read A=20            |
|             | unlock-s(A)          |
|             | commit               |

### 三级封锁协议

在二级的基础上，要去读取数据 A 时必须加 S 锁，直到事务结束后才能释放 S 锁。

由于事务读取数据 A 时，加了 S 锁，因此其他事务在该事物结束后才能加 X 锁，对数据 A 进行修改。确保了事务中每次读取数据 A 的结果一致。

| T1          | T2          |
| ----------- | ----------- |
| lock-s(A)   |             |
| read A=20   |             |
|             | lock-x(A)   |
|             | wait        |
| read A=20   |             |
| commit      |             |
| unlock-s(A) |             |
|             | obtain      |
|             | read A=20   |
|             | write A=19  |
|             | commit      |
|             | unlock-X(A) |

### 两段锁协议

加锁和解锁分为两个阶段，事务在解锁之后不能再加锁

事务遵循两段锁协议是保证并发操作可串行化调度的充分条件。例如以下操作满足两段锁协议，它是可串行化调度。

```
lock-x(A)...lock-s(B)...lock-s(c)...unlock(A)...unlock(C)...unlock(B)
```

但不是必要条件，例如以下操作不满足两段锁协议，但是它还是可串行化调度。

```
lock-x(A)...unlock(A)...lock-s(B)...unlock(B)...lock-s(c)...unlock(C)...
```



## 隔离级别

### 未提交读（READ UNCOMMITTED）

事务中的修改，即使没有提交，对其它事务也是可见的。

### 提交读（read committed）

一个事物只能读取已经提交的事务所做的修改。换句话说，一个事物所做的修改在提交之前对其它事务是不可见的。避免脏读。

### 可重复读（repeatable read）

保证在同一个事务中多次读取同样的数据的结果是一致的。避免不可重复读。

### 可串行化（serializable）

强制事务串行执行。

| 隔离级别 | 脏读 | 不可重复读 | 幻影读 |
| -------- | ---- | ---------- | ------ |
| 未提交读 | YES  | YES        | YES    |
| 提交读   | NO   | YES        | YES    |
| 可重复读 | NO   | NO         | YES    |
| 可串行化 | NO   | NO         | NO     |



## MVCC

多版本并发控制（Multi-Version Concurrency Control, MVCC）是MySQL 的 InnoDB 存储引擎实现隔离级别的一种具体方式，用于实现提交读和可重复读这两种隔离级别。而未提交读隔离界别总是读取最新的数据行，无须使用 MVCC；可串行化隔离级别需要对所有读取的行都加锁，单纯使用 MVCC 无法实现。

### 版本号

* 系统版本号：是一个递增的数字，每开始一个新的事务，系统版本号就会自动递增。
* 事务版本号：事务开始时的系统版本号。

InnoDB 的 MVCC 在每行记录后面都保存着这两个隐藏的列，用来存储两个版本号：

* 创建版本号：指示创建一个数据行的快照时的系统版本号；
* 删除版本号：如果该快照的删除版本号大于当前事务版本号表示该快照有效，否则表示该快照已经被删除了。

### Undo 日志

InnoDB 的 MVCC 使用到的快照存储在 Undo 日志中，该日志通过回滚指针把一个数据行的所有快照连接起来。
