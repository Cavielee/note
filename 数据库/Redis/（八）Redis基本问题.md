## Redis 与数据库一致性问题

Redis 作用是为了将 **热门数据** 或 **分布式数据** 从数据库缓存出来，从而减少数据库的压力。

当数据要更新时，此时 Redis 和 数据库（Mysql为例） 之间的数据可能存在不一致。

以下是一致性问题情境：

### 情景一

先更新数据库，再更新缓存

当线程①更新数据时，此时会先更新 Mysql 数据，然后再更新 Redis 缓存。然而线程②在线程①更新 Mysql 数据后，读取了 Redis 的旧缓存（脏数据）。

执行顺序如下：

```
线程1： ①更新Mysql数据					③更新Redis缓存
线程2： 				②读取Redis旧缓存
```



### 情景二

先更新数据库，再更新缓存

当线程①更新数据时，此时会先更新 Mysql 数据，然后再更新 Redis 缓存。然而线程②在线程①更新 Mysql 数据后，也去更新了该数据，使得最终线程①会把线程②的Redis修改覆盖掉。

执行顺序如下：

```
线程1： ①更新Mysql数据									④更新Redis缓存
线程2： 				②更新Mysql数据	③更新Redis缓存
```



### 情景三

先删除缓存，再更新数据库

当线程①更新数据，此时会先删除 Redis 中的缓存再更新 Mysql 的数据。然而线程②在线程①删除 Redis 的缓存后，去读取数据，此时会到 Mysql 中读取一个旧数据（脏数据），并更新到 Redis 上。

执行顺序如下：

```
线程1： ①删除Redis缓存											④更新Mysql数据
线程2： 			②读取Redis缓存（没有命中）	③读取Mysql旧数据
```



### 解决方案

1. 更新数据库，再删除缓存。

从上述多种场景来看，由于 Redis 和 Mysql 是在两个不同的操作，因此一定会存在一定的一致性问题。

而先更新数据库，再删除缓存的方案，是出现脏数据概率最低的。

```
线程1： ①更新Mysql数据					③删除Redis缓存	
线程2： 			②读取Redis缓存的旧数据
```



2. 只更新缓存，异步去更新数据库

把数据库更新操作放到消息队列中，并异步的去完成这些操作。

不能保持数据库强一致性。



## 缓存穿透

指黑客访问一些肯定不存在的 Key，从而缓存不会命中，落到了数据库查询。

结果：导致数据库压力增大。



### 解决方案

1. key 设置特殊的格式，如果请求不遵循该格式则无效。
2. 布隆过滤器。



## 缓存击穿

缓存击穿指某一时刻大量的 Key 同时过期，此时高并发的访问这些数据，所有请求都落在 DB 上，可能导致 DB 瞬时压力过大而崩溃。



### 解决方案

1. 互斥锁。当 Key 过期时，应当先获取分布式互斥锁，再去访问 DB，这样可以使访问同一资源的请求只有一个到达 DB 中。
2. 设计一个随机值。为每一个 key 设置一个随机的过期时间，使得这些 Key 不会在同一时刻同时过期。



## 缓存失效

类似于缓存击穿。某个时间 Key 失效。
